# E-Commerce Growth Analytics Platform

> **Status — Archived implementation.** This repository documents an end-to-end multi-channel DTC analytics pipeline originally built on **Microsoft Fabric** (Lakehouse + DirectLake) with **PySpark** transformations and a **Power BI** semantic model. The Fabric workspace trial license has since expired, so the dashboard is no longer live and the workspace cannot be re-entered without a new Azure subscription. **All code, architecture, and dashboard screenshots are preserved as a historical record.** For the full visual walkthrough of the original implementation see [`docs/original_fabric_implementation.md`](docs/original_fabric_implementation.md).

## Context

A simulated US-based DTC coffee concentrate brand (Javy-style) selling across three channels — **Shopify** (~3,000 orders/year), **Amazon FBA** (~5,000 orders/year), and **TikTok Shop** (~1,500 orders/year). The goal was to take the raw operational exports each channel produces (CSV / TSV / XLSX, all in slightly different shapes) and land them in a single warehouse model that finance and marketing could actually query.

All data in this repo is **synthetic**, generated by [`data/generate_data.py`](data/generate_data.py) using `Faker`. No real customer, order, or financial data is included. The simulation mirrors realistic patterns — seasonal lift in Nov–Dec, ~5% CAD orders on TikTok, ~0.5% duplicate Amazon orders, ~2% missing Shopify emails — so the cleaning logic exercised in the Silver layer is non-trivial.

## Original Architecture

The pipeline follows the **Medallion pattern** (Bronze → Silver → Gold) — an industry-standard approach for progressively refining data quality, popularized by Databricks and natively supported in Microsoft Fabric. The diagram below describes the architecture **as it was delivered**; it is not a "how to run this locally" guide.

```
┌──────────────┐   ┌──────────────┐   ┌──────────────┐
│   Shopify    │   │    Amazon    │   │  TikTok Shop │
│  Orders CSV  │   │  Orders TSV  │   │  Settlement  │
│              │   │  Settle TSV  │   │     XLSX     │
└──────┬───────┘   └──────┬───────┘   └──────┬───────┘
       │                  │                  │
       └──────────┬───────┴──────────┬───────┘
                  ▼                  ▼
        ┌────────────────────────────────────┐

        │    Raw files → Delta tables        │
```

## Key Metrics Surfaced

The Gold layer feeds a 10-KPI dashboard covering revenue, product, and channel health. One-line summary below; full formulas, grain, and source lineage live in [`docs/kpi_definitions.md`](docs/kpi_definitions.md).

- **GMV** — gross merchandise value per channel, daily/monthly
- **Net Revenue** — GMV minus platform fees, FBA, creator commissions, refunds
- **AOV** — average order value per channel (expected: Shopify $25–45, Amazon $18–35, TikTok $15–30)
- **Channel Revenue Mix** — Shopify / Amazon / TikTok contribution over time
- **Refund Rate** — by platform and SKU
- **Platform Fee Ratio** — total fees as % of GMV (Amazon ~15% referral + FBA; TikTok ~7% + 10–20% creator)
- **Top SKUs by Revenue** — cross-platform unified via SKU mapping
- **Repeat Purchase Rate** — Shopify-only (the only channel with customer email)
- **TikTok Creator ROI** — GMV-driven / commission-paid
- **Revenue Growth Rate (MoM)** — with expected Nov–Dec seasonal spike

## Engineering Decisions

### Medallion layering — why three layers, not one
The temptation on a project this size is to go Raw → Gold directly. The Medallion pattern was chosen deliberately because each layer solves a different problem: **Bronze** preserves an immutable copy of every source row so re-processing is always possible if business logic changes; **Silver** is where all the "messy" work happens (currency, dates, SKU naming, dedup) so the rest of the pipeline can assume clean inputs; **Gold** is the only layer that knows about business definitions (what "net revenue" means, what "cohort" means) and is the only one Power BI reads from. This separation pays for itself the first time a KPI definition changes — only the Gold table is rebuilt, Bronze and Silver are untouched.

### SKU mapping — 14 products × 3 channels with three naming conventions
Same physical product has a different SKU on each channel (Shopify uses `original-concentrate-6oz`, Amazon uses `JAVY-ORIG-6OZ`, TikTok uses `TS-ORIGINAL-6`). Without unification, every product-level metric (top SKU, refund rate by SKU) becomes meaningless. The Silver layer builds a `dim_products` table from a hand-maintained mapping (full table in [`docs/data_dictionary.md`](docs/data_dictionary.md)), and every fact table joins through it to expose a single `standardized_sku`.

### Amazon Settlement EAV pivot
The Amazon Settlement TSV arrives in **Entity-Attribute-Value** format: each fee component (Principal, Commission, FBA fee, Tax, Shipping) is a separate row keyed by `order-id`. For per-order net-margin analysis this has to be reshaped into a wide table — one row per order, one column per fee type. Silver does this with a PySpark `pivot()` on `amount-description`, then joins back to `fact_orders`. This is the single highest-value transformation in the pipeline: without it, net margin per order is uncomputable.

## Why This Is Preserved As-Is

The Fabric trial expired and a deliberate decision was made **not to rebuild on an open-source stack**. The reasoning:

- **Keyword coverage is already complete.** The 960 lines of PySpark across the three notebooks, the Medallion documentation, and the 9 visual artifacts already evidence every keyword this project is meant to demonstrate (PySpark / Fabric / Power BI / Medallion / Star Schema / Delta Lake / Azure). A DuckDB or local-Spark rebuild would not add a single new capability worth signaling.
- **A live dashboard would not survive long anyway.** Looker Studio or a self-hosted alternative would solve "dashboard is reachable" but introduce a new failure surface (free-tier quotas, OAuth re-auth, broken embeds) for marginal benefit.
- **CI/CD is already demonstrated elsewhere.** The sibling [`saas-product-analytics`](https://github.com/Leo309/saas-product-analytics) project carries the GitHub Actions / dbt tests / scheduled orchestration story. Duplicating it here would be redundant.
- **PySpark code cannot be exercised locally without significant rework.** The notebooks assume Fabric runtime helpers (`spark`, `mssparkutils`, Lakehouse-mounted paths). Re-shimming for local Spark is a non-trivial weekend on its own and produces nothing that wasn't already proven in Fabric.
